Nos últimos meses, tenho estado numa jornada para encontrar o editor de código perfeito que equilibre uma excelente experiência de desenvolvedor com o aumento de produtividade da assistência de IA. Esta exploração surgiu depois de eu já ter passado quase dois anos usando o VS Code, que adotei após abandonar Emacs e Neovim em busca de uma experiência de edição mais moderna. Minha busca mais recente me levou por vários editores modernos, principalmente aqueles da família VSCodium, com alguns desvios interessantes ao longo do caminho. Aqui está minha experiência e o que aprendi.

## O Ecossistema VSCodium: Uma Breve Visão Geral

Para aqueles que não estão familiarizados, os editores VSCodium são essencialmente forks do Visual Studio Code com foco em princípios de código aberto. Esses editores removem a telemetria e os elementos proprietários da Microsoft, mantendo a maioria dos recursos que tornam o VS Code popular. O ecossistema se expandiu recentemente com várias variantes aprimoradas por IA que se baseiam nessa fundação.

Esta família de editores compartilha uma ancestralidade comum, mas diverge em sua abordagem aos recursos, especialmente na integração de IA. Alguns membros notáveis incluem:

- **VS Code**: O editor original da Microsoft
- **VSCodium**: A versão verdadeiramente de código aberto
- **Cursor**: VS Code com assistência de codificação de IA aprimorada
- **Windsurf**: Outra variante focada em IA
- **GitPod**: Ambiente de desenvolvimento baseado em nuvem com base no VS Code

## Minha Jornada Começa: Cursor

Minha exploração começou com [Cursor](https://www.cursor.com/), que inicialmente me impressionou com sua elegante integração de capacidades de IA. Depois do VS Code, parecia uma evolução natural — familiar o suficiente para ser confortável, mas com superpoderes de IA adicionados.

Durante meu período de teste gratuito com o Cursor, descobri várias vantagens:

- **UI/UX Polida**: O Cursor manteve a experiência familiar do VS Code enquanto adicionava pontos de integração de IA bem pensados.
- **Assistência de IA Contínua**: A capacidade de obter complementos de código, explicações e sugestões de refatoração parecia natural e discreta.
- **Apelo Visual**: A estética geral do editor era agradável o suficiente para que retornar ao VS Code padrão depois parecesse um retrocesso.

A abordagem do Cursor à IA parecia menos intrusiva em comparação com alternativas como Cline ou Roo Code. Em vez de parecer um recurso complementar, as capacidades de IA no Cursor pareciam misturar-se naturalmente com o fluxo de trabalho de desenvolvimento.

## O Experimento Windsurf

Após o término do meu teste com o Cursor, decidi experimentar o [Windsurf](https://windsurf.com/editor), outro editor baseado em VS Code que prometia desenvolvimento aprimorado por IA. Infelizmente, essa experiência foi consideravelmente menos satisfatória:

- **Experiência Pouco Refinada**: O Windsurf carecia do refinamento que tornava o Cursor agradável de usar.
- **Integração de IA Inconveniente**: Os recursos de IA pareciam adicionados à força, em vez de integrados perfeitamente.
- **Problemas de Desempenho**: Encontrei atrasos ocasionais e problemas de estabilidade que interromperam meu fluxo de trabalho.

O forte contraste entre Cursor e Windsurf destacou a importância de uma implementação cuidadosa para ferramentas de desenvolvimento aprimoradas por IA. Simplesmente adicionar capacidades de IA não é suficiente — elas precisam parecer uma extensão natural do editor, e não um recurso secundário.

## O Problema das Extensões

Tanto o Cursor quanto o Windsurf (e o ecossistema VSCodium mais amplo) enfrentam um desafio significativo: o suporte a extensões. Este problema tornou-se ainda mais pronunciado após a recente repressão da Microsoft às variantes não oficiais do VS Code.

Como [Theo (T3.gg) discutiu em um vídeo recente](https://www.youtube.com/watch?v=vEQ07-p8ZDE), a Microsoft começou a restringir o acesso ao marketplace oficial do VS Code para clientes de terceiros. Isso significa que os usuários de editores baseados em VSCodium devem depender de fontes alternativas de extensões como o Open VSX, o que limita severamente suas opções.

O Open VSX Registry, embora tentando fornecer uma alternativa de código aberto ao marketplace da Microsoft, fica significativamente aquém em várias maneiras:

- **Seleção Limitada**: O Open VSX possui apenas uma fração das extensões disponíveis no marketplace oficial
- **Faltam Extensões Populares**: Muitas extensões amplamente utilizadas não foram portadas
- **Versões Desatualizadas**: Mesmo quando as extensões estão disponíveis, elas geralmente estão várias versões atrás de suas contrapartes oficiais
- **Qualidade Inconsistente**: Sem o mesmo processo de revisão, a qualidade das extensões pode ser imprevisível

Para muitos desenvolvedores, essas limitações se tornam obstáculos. Eu me vi sentindo falta de extensões cruciais, particularmente as específicas de linguagem e ferramentas de produtividade das quais eu havia passado a depender em meu fluxo de trabalho regular. Quando você está acostumado a ter um rico ecossistema de extensões para resolver problemas específicos, as limitações do Open VSX parecem particularmente restritivas.

## Retorno ao VS Code (Brevemente)

As limitações de extensões eventualmente me levaram de volta ao VS Code padrão, mas me vi sentindo falta da estética e das capacidades de IA do Cursor. Eu até tentei reconfigurar o VS Code para se parecer mais com o Cursor, o que ajudou um pouco, mas não replicou totalmente a experiência.

Este período de retorno ao VS Code me fez perceber o quanto eu valorizava ter a assistência de IA integrada diretamente à minha experiência de edição, em vez de mudar de contexto para uma janela de chat separada.

## Encontrando o Zed: Uma Alternativa Nova

Recentemente, tenho explorado o [Zed](https://zed.dev/), que oferece uma partida refrescante do paradigma do VS Code. Construído do zero com desempenho em mente, o Zed adicionou recentemente um robusto suporte a chat de IA que me impressionou.

O que faz o Zed se destacar:

- **Desempenho**: É visivelmente mais rápido e responsivo do que os editores baseados em VS Code.
- **Design Limpo**: A UI é minimalista, mas bem pensada, focando no código em vez de o cercar com painéis.
- **Capacidades de IA em Crescimento**: Embora mais recentes, os recursos de IA do Zed já são bastante úteis e estão melhorando rapidamente.
- **Abordagem Fresca**: Romper com o ecossistema do VS Code completamente oferece tanto desafios quanto oportunidades.

O Zed parece o início de algo novo, em vez de uma iteração sobre a tecnologia existente, o que é excitante e ocasionalmente limitante quando fluxos de trabalho específicos ainda não são suportados.

## Além dos Editores: Claude Desktop e MCP

Uma descoberta inesperada em minha jornada foi perceber que a IA integrada ao editor não é a única abordagem. Encontrei excelentes resultados usando o [Claude Desktop](https://www.anthropic.com/claude) com suas capacidades MCP (Machine Control Protocol) como um companheiro para minhas sessões de codificação.

Esta abordagem:

- Mantém meu editor limpo e focado na edição
- Fornece assistência de IA poderosa quando necessário
- Permite um prompt mais detalhado e configuração de contexto
- Funciona em qualquer editor que eu escolher usar

Embora não seja tão perfeitamente integrada quanto as soluções incorporadas, há algo a ser dito sobre a flexibilidade desta abordagem. Posso usar o editor que melhor se adapta às minhas necessidades enquanto ainda acesso uma poderosa assistência de IA.

## Fluxos de Trabalho Git: Entre o GitButler

Outro componente que adicionei à minha pilha de desenvolvimento é o [GitButler](https://gitbutler.com/), que uso em conjunto com o Zed. Em vez de depender apenas das ferramentas git integradas, o GitButler oferece uma abordagem mais visual e intuitiva para o gerenciamento de branches e alterações de código.

A combinação de Zed para edição e GitButler para controle de versão me dá o melhor dos dois mundos:

- Experiência de edição limpa e focada
- Poderoso fluxo de trabalho git visual
- Liberdade de não estar preso à abordagem de git de uma única ferramenta

## Conclusão: Estado Atual e Perspectivas Futuras

Por enquanto, continuo com o Zed como meu editor principal, complementado pelo Claude Desktop para assistência de IA e pelo GitButler para gerenciamento de git. Esta combinação parece flexível o suficiente para evoluir à medida que as ferramentas melhoram, ao mesmo tempo em que oferece um ambiente produtivo hoje.

Dito isso, não vou mentir — às vezes usar o GitButler parece um pouco complicado porque tenho que trocar de janela apenas para fazer um commit. Mesmo com as muitas vantagens de ter uma interface git dedicada separada, às vezes parece que meu fluxo de trabalho é mais complicado do que deveria ser. Este é o preço de usar ferramentas especializadas em vez de uma solução tudo-em-um.

O que aprendi com esta jornada:

1. O editor certo é altamente pessoal e depende do seu fluxo de trabalho específico
2. A assistência de IA pode ser transformadora, mas a implementação importa muito
3. A saúde do ecossistema de extensões é crucial para a viabilidade a longo prazo, quero dizer, elas realmente resolvem muitos problemas para muitas pessoas - o Open VSX está longe de ser útil para mim
4. Às vezes, a melhor solução é uma combinação de ferramentas especializadas em vez de um único editor que faz tudo — mas isso vem com seus próprios pontos de atrito

Suspeito que esta jornada não terminou, pois o cenário das ferramentas de desenvolvimento continua a evoluir rapidamente, especialmente com a integração de IA se tornando cada vez mais sofisticada. Mas, por enquanto, encontrei uma configuração que funciona bem para minhas necessidades, permanecendo aberto a novas possibilidades.
